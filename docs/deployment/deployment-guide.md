# wAether Deployment Guide

This guide covers the complete deployment process for the wAether Android Wear OS application, from development setup through production release.

## Prerequisites

### Development Environment

- **Android Studio**: Latest stable version (Hedgehog 2023.1.1 or later)
- **JDK**: OpenJDK 17 or later
- **Android SDK**: API 30+ (Android 11+) for Wear OS
- **Git**: Version control system
- **Firebase CLI**: For Firebase configuration and deployment

### Required Accounts and Services

1. **Google Developer Account**: For Play Store publishing
2. **Firebase Project**: For backend data storage
3. **GitHub Account**: For source code management (if using CI/CD)

## Initial Setup

### 1. Environment Configuration

```bash
# Clone the repository
git clone https://github.com/GooseyPrime/wAether.git
cd wAether

# Set up environment variables
cp .env.template .env
# Edit .env with your configuration (see README_ENV.md)
```

### 2. Firebase Setup

1. **Create Firebase Project**:
   ```bash
   # Install Firebase CLI
   npm install -g firebase-tools
   
   # Login to Firebase
   firebase login
   
   # Initialize project
   firebase init
   ```

2. **Configure Firestore**:
   - Enable Firestore Database in Firebase Console
   - Set up security rules (see `docs/production/firebase-security-rules.md`)
   - Configure indexes if needed

3. **Download Configuration**:
   - Download `google-services.json` from Firebase Console
   - Place in `app/` directory
   - Verify file is in `.gitignore`

### 3. API Configuration

1. **OpenMeteo API**:
   - No setup required for basic usage
   - Register at [openmeteo.com](https://openmeteo.com/) for higher limits

2. **NOAA Space Weather API**:
   - No setup required (public API)
   - Verify access at [services.swpc.noaa.gov](https://services.swpc.noaa.gov)

## Build Process

### Development Build

```bash
# Make gradlew executable (if needed)
chmod +x ./gradlew

# Clean build
./gradlew clean

# Debug build
./gradlew assembleDebug

# Install on connected device
./gradlew installDebug
```

### Release Build

```bash
# Generate release build
./gradlew assembleRelease

# Generate signed APK (requires keystore configuration)
./gradlew bundleRelease
```

### Build Variants

| Variant | Purpose | Configuration | Debugging |
|---------|---------|---------------|-----------|
| `debug` | Development | Debug enabled, verbose logging | Full debugging |
| `release` | Production | Optimized, minimal logging | Limited debugging |

## Code Signing

### Debug Signing

Debug builds use the default debug keystore automatically generated by Android Studio.

### Release Signing

1. **Generate Release Keystore**:
   ```bash
   keytool -genkey -v -keystore waether-release-key.keystore \
     -alias waether -keyalg RSA -keysize 2048 -validity 10000
   ```

2. **Configure Signing in `app/build.gradle.kts`**:
   ```kotlin
   android {
       signingConfigs {
           release {
               storeFile file("path/to/waether-release-key.keystore")
               storePassword "STORE_PASSWORD"
               keyAlias "waether"
               keyPassword "KEY_PASSWORD"
           }
       }
       buildTypes {
           release {
               signingConfig signingConfigs.release
               // ... other configuration
           }
       }
   }
   ```

3. **Security Notes**:
   - Store keystore file securely (not in version control)
   - Use environment variables for passwords in CI/CD
   - Keep backup of keystore and passwords
   - Document keystore recovery procedures

## Continuous Integration/Continuous Deployment (CI/CD)

### GitHub Actions Workflow

Create `.github/workflows/android.yml`:

```yaml
name: Android CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v4
    
    - name: Set up JDK 17
      uses: actions/setup-java@v4
      with:
        java-version: '17'
        distribution: 'temurin'
        
    - name: Setup Gradle
      uses: gradle/gradle-build-action@v2
      
    - name: Grant execute permission for gradlew
      run: chmod +x gradlew
      
    - name: Run tests
      run: ./gradlew test
      
    - name: Build debug APK
      run: ./gradlew assembleDebug
      
    - name: Upload build artifacts
      uses: actions/upload-artifact@v3
      with:
        name: debug-apk
        path: app/build/outputs/apk/debug/
```

### Deployment Pipeline Stages

1. **Source Control**: Code committed to repository
2. **Build**: Automated build and test execution
3. **Test**: Unit and instrumented tests
4. **Code Quality**: Linting and static analysis
5. **Security Scan**: Dependency and vulnerability scanning
6. **Staging Deploy**: Deploy to staging environment
7. **Manual Testing**: QA verification
8. **Production Deploy**: Release to production

## Production Deployment

### Google Play Store Deployment

1. **Prepare Release**:
   ```bash
   # Generate signed bundle
   ./gradlew bundleRelease
   ```

2. **Upload to Play Console**:
   - Upload AAB file to Google Play Console
   - Complete store listing information
   - Set up app pricing and distribution
   - Configure release rollout (staged rollout recommended)

3. **Release Management**:
   - Use staged rollout (start with 1-5% of users)
   - Monitor crash reports and user feedback
   - Gradually increase rollout percentage
   - Full rollout after validation

### Rollback Procedures

1. **Emergency Rollback**:
   - Halt current rollout in Play Console
   - Return to previous stable version
   - Investigate and fix issues

2. **Planned Rollback**:
   - Document rollback decision and reasons
   - Notify stakeholders
   - Execute rollback during maintenance window
   - Update monitoring and alerts

## Troubleshooting

### Common Deployment Issues

**Build Failures**:
- Check Android SDK and build tools versions
- Verify Gradle and plugin compatibility
- Clean and rebuild: `./gradlew clean build`

**Signing Issues**:
- Verify keystore path and credentials
- Check key alias and validity period
- Ensure consistent signing configuration

**Firebase Configuration**:
- Verify `google-services.json` is current
- Check Firebase project permissions
- Validate Firestore security rules

**Play Store Rejections**:
- Review Play Console policy compliance
- Check app permissions and justifications
- Verify target API level requirements
